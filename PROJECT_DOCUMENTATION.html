🎯 Project Goal

PRIMARY OBJECTIVE:

Build a production-ready machine learning system that predicts which software modules are likely to have defects, enabling developers and QA teams to prioritize testing efforts and allocate resources more effectively.

Why This Matters

Software defects are expensive. The later a bug is discovered in the development cycle, the more it costs to fix. By predicting which modules are likely to have defects early, teams can:

  • Reduce Testing Costs: Focus QA efforts on high-risk modules
  • Improve Code Quality: Identify problem areas before production
  • Accelerate Development: Spend less time on low-risk modules
  • Make Data-Driven Decisions: Use ML predictions instead of gut feeling

Business Problem

Given a software module with metrics like:

  • Lines of Code (LOC)
  • Cyclomatic Complexity
  • Halstead Metrics
  • Code Structure Information

Predict: Will this module have defects? If yes, what's the risk level?

📦 Core Components

1. Data Preprocessing Module

Input: Raw software metrics (21+ features)
Steps:
• Handle missing values (median imputation)
• Remove exact duplicates
• Encode target variable (True→1, False→0)
• Remove zero-variance columns
• StandardScaler normalization
• SMOTE resampling (if class imbalance)
• 80/20 stratified train/test split
Output: Normalized, balanced training data ready for ML

2. EDA Module

Visualizations Generated:
• Class Distribution (bar chart with %)
• Correlation Heatmap (full matrix)
• Feature Histograms (by defect class)
• Boxplots (top 8 correlated features)
• Statistical Feature Importance (SelectKBest)
• Outlier Detection (LOC, v(g), ev(g), n)
Statistical Reports:
• Descriptive statistics (mean, std, min, max)
• Skewness and Kurtosis per feature
• Pearson correlation with target

3. ML Models Module

7 Models Trained & Evaluated:
• Decision Tree (depth=5)
• Random Forest (100 estimators)
• SVM (RBF kernel)
• Logistic Regression (max_iter=1000)
• Naive Bayes (Gaussian)
• XGBoost (gradient boosting)
• LightGBM (fast gradient boosting)
For Each Model:
• Trained on SMOTE-balanced data
• Evaluated on original test set
• 6 metrics computed (Accuracy, Precision, Recall, F1, ROC-AUC, CM)
• Classification report printed

4. Decision Support System (DSS)

Features:
• Risk Score Calculation (0-100 scale)

• Module Ranking Table (sorted by risk)
• Alert System (CRITICAL modules flagged)
• Recommendations Engine (context-aware)
• QA Priority Queue (ordered testing list)

5. Streamlit Dashboard

6 Interactive Tabs:

🤖 Machine Learning Models

Model Selection Rationale

Different models capture different patterns in the data. By training 7 models with different algorithms, we ensure robustness and can select the best performer for the specific task.

Model Type Key Strengths Use Case
Decision Tree Tree-based Interpretable, no scaling needed Baseline, rule extraction
Random Forest Ensemble Robust, feature importance, parallel Strong baseline, production
SVM Kernel-based Effective in high dimensions Complex non-linear patterns
Logistic Regression Linear Fast, probabilistic, interpretable Baseline, production
Naive Bayes Probabilistic Fast training, good for imbalanced Fast predictions, baseline
XGBoost Gradient Boosting State-of-art, handles imbalance Best-in-class performance
LightGBM Gradient Boosting Fast, memory efficient, scalable Large datasets, production

Hyperparameter Tuning

GridSearchCV applied to top 2 models (based on Recall score)
Cross-validation: 3-Fold during grid search
Scoring metric: Recall (prioritize finding defects)
Parameters searched: n_estimators, max_depth, learning_rate, C, gamma

Cross-Validation

Strategy: 5-Fold Stratified K-Fold
Applied to: Best model (highest Recall)
Metrics: Mean and Std for Accuracy, Precision, Recall, F1, ROC-AUC
Purpose: Ensure model generalization and robustness

🎁 Bonus Features

Association Rules Mining

Discovers patterns in software metrics that lead to defects:

Example Rule: {complexity=High, modifications=High} → {defects=High}

• Confidence: 85% (85% of modules with this pattern had defects)
• Lift: 2.3 (2.3x more likely to have defects)

Clustering Analysis

Groups similar modules to identify risk zones:

  • DBSCAN: Density-based clustering identifies outliers
